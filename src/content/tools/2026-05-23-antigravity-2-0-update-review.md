# 안티그래비티 2.0 업데이트 후기 - AI 에이전트는 더 똑똑해졌지만 더 조심해야 한다

구글 안티그래비티가 최근 2.0으로 업데이트됐습니다.

공식 블로그 기준으로 안티그래비티 2.0은 2026년 5월 19일 공개됐고, 기존보다 더 에이전트 중심의 작업 환경을 강조하고 있습니다. 독립형 데스크톱 앱, CLI, SDK, 서브 에이전트, 예약 작업 같은 기능이 함께 소개됐습니다.

쉽게 말하면 이제 안티그래비티는 단순히 “코드 고쳐주는 AI 도구”라기보다, 여러 작업을 AI 에이전트에게 맡기고 관리하는 방향으로 더 강해진 느낌입니다.

저도 안티그래비티를 계속 사용하고 있었기 때문에 이번 업데이트가 꽤 크게 느껴졌습니다.

좋아진 부분도 분명히 있습니다.  
그런데 동시에 더 조심해야겠다는 생각도 들었습니다.

AI 에이전트가 강해졌다는 건, 잘 쓰면 훨씬 빠르게 일을 처리할 수 있다는 뜻입니다.  
하지만 반대로 잘못 시키면 더 넓은 범위에서 엉뚱한 파일을 건드릴 수도 있다는 뜻이기도 합니다.

이번 글에서는 안티그래비티 2.0 업데이트 이후 실제로 써보며 느낀 점을 정리해보겠습니다.

## 안티그래비티 2.0은 뭐가 달라졌을까

안티그래비티 2.0의 핵심은 “에이전트 중심”입니다.

예전에는 코드 에디터 안에서 AI에게 도움을 받는 느낌이 강했습니다.  
사용자가 파일을 보고, AI에게 물어보고, 필요한 부분을 수정하는 흐름이었습니다.

그런데 2.0은 조금 더 적극적인 방향으로 바뀐 느낌입니다.

AI가 파일을 읽고, 작업을 나누고, 필요한 명령을 실행하고, 결과를 보고하는 흐름이 더 강해졌습니다.

공식 발표에서도 안티그래비티 2.0은 독립형 데스크톱 앱과 에이전트 관리 기능을 강조하고 있습니다. 여기에 CLI, SDK, 서브 에이전트, 예약 작업 같은 기능도 함께 소개됐습니다.

이 말은 단순히 기능이 많아졌다는 뜻만은 아닙니다.

AI가 한 번에 맡을 수 있는 작업 범위가 더 넓어졌다는 뜻입니다.

파일 하나를 고치는 수준을 넘어, 여러 파일을 읽고, 구조를 파악하고, 작업을 나눠서 처리하는 방향으로 가고 있습니다.

좋게 말하면 훨씬 강력해졌습니다.  
나쁘게 말하면, 잘못 지시했을 때 더 크게 꼬일 수도 있습니다.

## AI가 강해질수록 지시가 더 중요해진다

안티그래비티를 쓰면서 가장 크게 느낀 건 이것입니다.

AI가 똑똑해질수록, 사용자의 지시도 더 정확해야 합니다.

기존 챗봇은 질문에 답만 했습니다.

“이 코드 왜 오류야?”라고 물으면 설명을 해줍니다.  
“블로그 글 어떻게 써?”라고 물으면 예시를 줍니다.

하지만 안티그래비티 같은 에이전트형 도구는 답만 하지 않습니다.

파일을 직접 열고, 코드를 수정하고, 명령어를 실행하고, 결과를 확인합니다.

여기서 문제가 생깁니다.

사용자는 “이것만 해줘”라고 생각하고 말했는데, AI는 “이것도 같이 해야겠네”라고 판단할 수 있습니다.

예를 들어 이런 경우입니다.

“블로그 글 하나 추가해줘.”

사용자는 새 글 파일만 추가되길 원합니다.

그런데 AI는 글 목록도 업데이트해야 한다고 생각할 수 있습니다.  
사이트맵이나 피드 파일까지 확인하려고 할 수도 있습니다.  
관련 카테고리 파일을 건드릴 수도 있습니다.

사용자 입장에서는 갑자기 일이 커집니다.

“아니, 누가 거기까지 하랬어?”

이 상황이 생기는 이유는 AI가 아무것도 못해서가 아닙니다.  
오히려 너무 적극적으로 해결하려고 하기 때문에 생기는 문제입니다.

## 좋아진 점 1. 작업 속도는 확실히 빠르다

안티그래비티를 쓰는 가장 큰 이유는 속도입니다.

코딩을 잘 모르는 사람이 직접 파일 구조를 찾고, 코드 위치를 찾고, 오류를 해결하려면 시간이 오래 걸립니다.

그런데 안티그래비티는 한국어로 설명해도 꽤 잘 알아듣습니다.

예를 들면 이런 식입니다.

“블로그 글 목록에서 5월 23일 글이 안 보이는 이유를 확인해줘.”

“이 파일은 수정하지 말고 원인만 알려줘.”

“새 글 파일만 추가하고, 기존 파일은 건드리지 마.”

이렇게 지시하면 관련 파일을 찾아보고, 원인을 추정하고, 필요한 작업을 제안합니다.

물론 항상 완벽하진 않습니다.

그래도 사람이 처음부터 하나씩 뒤지는 것보다 훨씬 빠릅니다.

특히 저처럼 코딩을 전문적으로 배운 사람이 아니라면, 이 속도 차이가 크게 느껴집니다.

예전 같으면 검색하고, 강의 보고, 파일 열어보고, 오류 메시지 보면서 한참 헤맸을 일을 AI가 먼저 찾아줍니다.

이건 분명히 큰 장점입니다.

## 좋아진 점 2. 혼자 사이트 운영할 때 도움이 된다

혼자 사이트를 운영하다 보면 생각보다 해야 할 일이 많습니다.

글도 써야 하고,  
이미지도 넣어야 하고,  
링크도 확인해야 하고,  
오류도 고쳐야 하고,  
검색 노출 문제도 신경 써야 합니다.

처음에는 글 하나 올리는 일처럼 보여도, 실제로는 연결된 작업이 많습니다.

안티그래비티 같은 에이전트 도구는 이런 반복 작업에서 도움이 됩니다.

예를 들어 블로그 글 파일을 만들고, 형식을 맞추고, 날짜나 카테고리를 확인하고, 빌드 오류를 찾는 일은 사람이 직접 하면 꽤 번거롭습니다.

하지만 AI 에이전트에게 맡기면 훨씬 빠르게 처리할 수 있습니다.

특히 정보성 사이트나 도구 모음 사이트처럼 계속 업데이트해야 하는 사이트에서는 이런 차이가 더 크게 느껴집니다.

한 번 만들고 끝나는 사이트보다, 계속 관리해야 하는 사이트에서 안티그래비티 같은 도구의 가치가 더 큽니다.

## 아쉬운 점 1. 여전히 자기 멋대로 움직일 때가 있다

하지만 좋은 점만 있는 건 아닙니다.

안티그래비티를 쓰면서 가장 답답한 부분은 여전히 “시키지 않은 일을 할 때가 있다”는 점입니다.

예를 들어 저는 분명히 이렇게 말합니다.

“이 파일만 수정해.”  
“기존 문장은 바꾸지 마.”  
“확인만 하고 수정하지 마.”  
“git push는 하지 마.”

그런데도 가끔은 다른 파일을 건드리거나, 문장을 다듬거나, 수정까지 진행하려고 할 때가 있습니다.

이건 AI 에이전트 도구를 쓸 때 가장 조심해야 하는 부분입니다.

AI는 사용자의 목적을 해결하려고 합니다.  
그런데 이 과정에서 “내가 보기엔 이것도 같이 고쳐야 한다”고 판단해버릴 수 있습니다.

사람 입장에서는 그게 친절이 아니라 사고입니다.

특히 운영 중인 사이트에서는 작은 수정 하나가 검색 노출, URL, 카테고리, 디자인에 영향을 줄 수 있습니다.

그래서 안티그래비티가 아무리 편해도, 결과를 그대로 믿고 바로 배포하면 위험합니다.

## 아쉬운 점 2. 완료했다고 했는데 실제로는 빠뜨릴 때가 있다

또 하나 조심해야 할 부분은 완료 보고입니다.

AI가 “작업 완료했습니다”라고 말했는데, 실제로 확인해보면 빠진 부분이 있을 때가 있습니다.

예를 들어 글 파일은 만들었지만 목록에 안 뜬다거나,  
수정했다고 했는데 실제 파일은 그대로라거나,  
오류를 해결했다고 했는데 빌드하면 다시 오류가 나는 경우가 있습니다.

이럴 때는 정말 허탈합니다.

하지만 아직은 AI 에이전트를 쓸 때 감수해야 하는 부분인 것 같습니다.

AI의 완료 보고는 “검수 끝”이 아닙니다.

정확히 말하면 “AI 기준으로는 끝났다고 판단했다”에 가깝습니다.

그래서 반드시 사람이 직접 확인해야 합니다.

작업 후에는 최소한 아래 정도는 확인하는 게 좋습니다.

1. 어떤 파일이 수정됐는지
2. 요청하지 않은 파일이 바뀌었는지
3. 빌드나 실행 결과가 정상인지
4. 실제 화면에서 원하는 대로 보이는지
5. git push 전에 변경 사항이 맞는지

이걸 확인하지 않으면 나중에 엉뚱한 부분이 바뀐 걸 발견할 수 있습니다.

## 안티그래비티를 쓸 때 가장 중요한 습관

제가 안티그래비티를 쓰면서 배운 가장 중요한 습관은 이것입니다.

“작업을 크게 맡기지 말고, 작게 쪼개서 맡기기.”

예를 들어 이렇게 말하면 위험합니다.

“사이트 오류 전체적으로 고쳐줘.”

이건 너무 넓습니다.

AI가 어떤 파일을 어디까지 고칠지 알 수 없습니다.  
디자인을 건드릴 수도 있고, 기존 구조를 바꿀 수도 있고, 의도하지 않은 부분까지 수정할 수 있습니다.

대신 이렇게 말하는 게 낫습니다.

“모바일 메뉴에서 X 버튼이 2개 보이는 문제만 확인해줘. 지금은 수정하지 말고 원인만 알려줘.”

또는 이렇게 말할 수 있습니다.

“블로그 글 목록에 2026년 5월 23일 글이 안 보이는 이유만 확인해줘. 기존 파일은 절대 수정하지 마.”

이렇게 범위를 좁히면 사고가 줄어집니다.

AI에게 일을 잘 시키는 건 결국 “명령을 잘게 나누는 일”입니다.

## 확인만 할 때 쓰는 지시문

안티그래비티를 쓰면서 저는 확인 작업과 수정 작업을 꼭 나누는 편이 좋다고 느꼈습니다.

확인만 시킬 때는 이렇게 쓰면 좋습니다.

지금은 확인만 해줘.  
절대 파일을 수정하지 마.  
문제 원인만 찾아서 알려줘.  
수정이 필요하면 어떤 파일을 어떻게 바꿔야 하는지만 설명해줘.  
내가 허락하기 전까지 수정하지 마.

이렇게 적어야 AI가 바로 수정에 들어가는 일을 줄일 수 있습니다.

물론 이렇게 써도 100% 완벽하게 막을 수 있는 건 아닙니다.

그래도 그냥 “확인해줘”라고 쓰는 것보다는 훨씬 안전합니다.

## 수정까지 맡길 때 쓰는 지시문

수정까지 맡길 때도 범위를 좁게 적는 게 좋습니다.

예를 들면 이렇게 말할 수 있습니다.

아래 작업만 수정해줘.

목표:  
2026년 5월 23일 글이 블로그 목록에 보이도록 수정

수정 가능 범위:  
글 목록을 불러오는 파일만 수정 가능

금지사항:  
기존 글 제목, URL, 카테고리, 날짜는 수정하지 마.  
디자인은 바꾸지 마.  
요청하지 않은 리팩토링은 하지 마.  
새로운 기능을 추가하지 마.  
git push는 하지 마.

작업 후 보고:  
수정한 파일 목록을 알려줘.  
각 파일에서 바꾼 내용을 요약해줘.  
요청하지 않은 변경이 있었는지 알려줘.

이렇게 적으면 AI가 해야 할 일과 하지 말아야 할 일이 분명해집니다.

안티그래비티가 강력해질수록 이런 지시문이 더 중요해집니다.

## 안티그래비티 2.0은 누구에게 좋을까

안티그래비티 2.0은 특히 이런 사람에게 잘 맞는 도구라고 느꼈습니다.

코딩은 잘 모르지만 사이트를 만들고 싶은 사람.  
블로그나 정보성 사이트를 직접 운영하는 사람.  
반복되는 파일 수정이 많은 사람.  
AI에게 코드 작업을 맡겨보고 싶은 사람.  
혼자 여러 개의 사이트나 프로젝트를 관리하는 사람.

저도 코딩을 전문적으로 배운 사람은 아닙니다.

그런데 안티그래비티 덕분에 사이트를 만들고, 글을 추가하고, 오류를 찾아보고, 여러 작업을 시도할 수 있었습니다.

이건 예전 같으면 쉽게 하기 어려운 일이었습니다.

그래서 안티그래비티 같은 도구는 분명히 큰 기회입니다.

다만 “AI가 다 알아서 해주니까 나는 안 봐도 된다”는 식으로 접근하면 위험합니다.

오히려 반대입니다.

AI가 일을 빨리 해주는 만큼, 사람은 더 정확하게 지시하고 더 꼼꼼하게 확인해야 합니다.

안티그래비티 2.0은 확실히 더 강력해졌습니다.

단순한 코드 보조 도구라기보다, AI 에이전트를 중심으로 작업을 맡기고 관리하는 방향으로 발전하고 있습니다.

코딩을 잘 모르는 사람에게는 분명히 좋은 도구입니다.  
혼자 사이트를 만들고 운영하는 사람에게도 큰 도움이 됩니다.

하지만 강력해진 만큼 더 조심해야 합니다.

AI가 파일을 직접 수정할 수 있다는 건 편리한 일이지만, 동시에 위험한 일이기도 합니다.

시키지 않은 파일을 건드릴 수 있고,  
완료했다고 말해놓고 빠뜨릴 수 있고,  
사용자가 원하지 않은 방향으로 코드를 바꿀 수도 있습니다.

그래서 안티그래비티를 쓰면서 세 가지를 꼭 기억해야 합니다.

첫째, 작업을 작게 나눠서 시키기.  
둘째, 하지 말아야 할 일을 분명히 적기.  
셋째, 완료 보고를 믿기 전에 직접 확인하기.

AI 에이전트는 앞으로 더 강력해질 겁니다.

하지만 지금 단계에서는 완전히 맡겨두는 도구라기보다, 빠르게 실행해주는 조수에 가깝습니다.

좋은 조수를 제대로 쓰려면 지시가 정확해야 합니다.

안티그래비티 2.0을 써보면서 가장 크게 느낀 건 이것입니다.
